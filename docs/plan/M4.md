# M4 — PagedAttention：从固定 slot 到分页 KV Cache

> **状态**：⬜ 未开始
> **依赖**：M3 — Continuous Batching
> **主层**：L2 Memory
> **知识文档**：[m4-paged-attention.md](../knowledge/m4-paged-attention.md)
> **任务卡目录**：[M4-archive/](../tasks/M4-archive/)

---

## 0. TL;DR

M3 已经解决了多请求如何在 decode iteration 边界重新组 batch，但 KV Cache 仍是固定 slot：每个 running 请求独占一个 `[max_seq_len]` 长度的连续物理空间。M4 要把这个连续空间改成分页管理：请求看到的是连续逻辑 token，底层映射到一组不连续的物理 block。

M4 的目标不是追求 vLLM 级性能，而是手写一个**可读、可测、可解释的 PyTorch 伪版 PagedAttention**，为 M5 Prefix Cache 和 M9 Triton kernel 打地基。

---

## 1. 背景：M3 fixed-slot 的问题

M3 的 `BatchedKVCache` 布局：

```text
layers[layer].k: [max_num_slots, n_kv_heads, max_seq_len, head_dim]
```

这很直观，但有三个问题：

| 问题 | M3 表现 | M4 目标 |
|---|---|---|
| 内存浪费 | 每个请求预留 `max_seq_len`，短请求也占满一个 slot | 只按实际长度分配 block，浪费最多一个 block |
| 无共享 | 两个请求即使 prompt 相同，也各自占一份 KV | 通过 block table + refcount 为 M5 prefix cache 准备 |
| 无 CoW | 不能表达多个请求共享前缀后分叉 | 支持 refcount + Copy-on-Write |

vLLM 的 PagedAttention 论文/博客指出，传统 KV 管理会因碎片和过度预留浪费 60%–80% 显存；PagedAttention 将每个序列的 KV 切成固定大小 block，连续逻辑块映射到非连续物理块，浪费只发生在最后一个 block，实践中低于 4%。

## 2. 目标与非目标

### 2.1 目标

| 能力 | M4 是否做 | 说明 |
|---|---|---|
| 物理 KV block 池 | ✅ | `[num_blocks, block_size, n_kv_heads, head_dim]` |
| per-request block table | ✅ | `request_id -> list[block_id]` |
| 按需分配 block | ✅ | prefill/decode 时按长度增长 |
| refcount | ✅ | 物理 block 引用计数 |
| Copy-on-Write | ✅ | 共享 block 写入前复制 |
| PyTorch 伪版 paged attention | ✅ | gather 出临时连续 KV，再复用现有 attention 逻辑 |
| M3 batch_generate 集成 | ✅ | 用 PagedKVCache 替换 BatchedKVCache 路径 |
| 正确性验证 | ✅ | fixed-slot vs paged token/logits 对齐 |

### 2.2 非目标

| 不做 | 原因 | 后续 |
|---|---|---|
| Triton paged attention kernel | M4 聚焦机制，不死磕 kernel | M9 |
| FlashAttention block table 原生调用 | MPS/纯 PyTorch 教学环境不适合 | M9 |
| Prefix Cache 命中复用 | M4 只准备 refcount/CoW 基础设施 | M5 |
| Chunked Prefill | 属于长上下文调度 | M10 |
| 复杂 preemption | 暂不引入抢占策略 | 后续 |
| CUDA Graph | 工程优化，不属于机制学习 | Release/M9 |

## 3. 核心设计

### 3.1 数据结构

```text
PagedKVCache
├── layers: list[PagedLayerKVCache]
│   ├── k: [num_blocks, block_size, n_kv_heads, head_dim]
│   └── v: [num_blocks, block_size, n_kv_heads, head_dim]
├── block_manager: BlockManager
└── request_tables: dict[str, BlockTable]

BlockTable
├── request_id: str
├── block_ids: list[int]      # logical block idx -> physical block id
└── seq_len: int              # 当前有效 token 数

BlockManager
├── free_block_ids: deque[int]
├── ref_counts: torch.Tensor / list[int]
└── block_size: int
```

### 3.2 逻辑地址到物理地址

给定请求的 token 位置 `pos`：

```text
logical_block = pos // block_size
block_offset  = pos % block_size
physical_block = block_table.block_ids[logical_block]
physical index = (physical_block, block_offset)
```

### 3.3 M4 与 M3 的接口关系

M3 decode 使用：

```python
cache_slots = torch.tensor([req.slot_id for req in running])
cache_positions = cache.seq_lens[cache_slots]
model(..., kv_cache=cache, cache_slots=cache_slots, cache_positions=cache_positions)
```

M4 改成：

```python
request_ids = [req.request_id for req in running]
cache_positions = paged_cache.positions(request_ids)
block_tables = paged_cache.block_tables(request_ids)
model(..., kv_cache=paged_cache, request_ids=request_ids, cache_positions=cache_positions)
```

为了降低改动，M4 可以保留 `cache_slots` 参数名一段时间，但语义会从 slot id 迁移为 request/block table 索引。任务卡中需要明确接口命名，避免 M3 语义残留。

## 5. 测试策略

### 5.1 L0 单测

- `test_block_manager.py`
  - allocate/free 基本功能
  - block 耗尽报错
  - refcount inc/dec 守恒
  - CoW：refcount>1 时写前复制

- `test_block_table.py`
  - `pos -> (block_id, offset)` 映射
  - append token 跨 block 边界
  - 多请求 block table 独立

- `test_paged_kv_cache.py`
  - prefill 写入多 block
  - decode 追加写入
  - gather 后与连续 KV 等价

### 5.2 L1/L2 正确性

- 单请求：M2/M3 fixed-slot 与 M4 paged logits 对齐。
- 多请求：M3 `batch_generate` fixed-slot 与 M4 paged 输出 token 级等价。
- 变长请求：不同 prompt/output 长度跨 block 边界。

### 5.3 Benchmark

M4 benchmark 不以速度为第一目标，主要看：

| 指标 | 预期 |
|---|---|
| memory blocks allocated | 接近实际 token 数 / block_size |
| internal fragmentation | 最多每请求浪费 `block_size-1` tokens |
| throughput | 可能比 M3 更慢（PyTorch gather 更复杂），接受 |

## 7. 风险与决策

| 风险 | 决策 |
|---|---|
| PyTorch 伪版性能更慢 | 接受；M4 是机制教学，性能留 M9 |
| block table 接口污染 M3 | 通过单独 `PagedKVCache` 类型分派，尽量不破坏 `BatchedKVCache` |
| CoW 没有 Prefix Cache 场景难测 | T1/T2 用人工共享 block table 构造测试 |
| block_size 选多大 | 默认 16 或 32，便于单测跨 block；不沿用 nano-vllm 的 256 |
| M4 与 M5 边界不清 | M4 做共享机制，不做 hash prefix lookup；M5 做 prefix cache policy |
