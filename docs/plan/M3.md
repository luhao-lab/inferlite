# M3 — Continuous Batching：多请求 Decode 调度

> **状态**：规划中
> **前置**：M2 — 单序列静态 KV Cache
> **主战场**：L3 Engine / Scheduler
> **目标**：在 M2 的单请求 KV Cache 基础上，引入教学版 continuous batching：请求可在 decode 迭代边界动态进出 batch，多请求共享一次模型 forward，从而提升聚合吞吐。

## 1. 技术调研：Continuous Batching 的来源与本质

### 1.1 问题背景：传统 batching 不适合自回归生成

LLM 生成是多轮迭代过程：一个请求不是一次 forward 完成，而是每生成一个 token 都要再 forward 一次。

传统 static batching 的问题是：batch 一旦形成，就要等整个 batch 全部请求完成后才能释放；短请求已经结束也不能返回，新请求也不能加入，只能等当前 batch 全部完成。

Orca OSDI'22 明确指出，传统 inference serving 不适合这类 multi-iteration workload，因为当前 batch 不能动态改变：早完成请求不能及时返回，新请求必须等当前 batch 完全结束；Orca 提出的 iteration-level scheduling 把调度粒度从“请求级”改为“迭代级”，每次只执行 batch 的一个模型迭代。[^orca]

FriendliAI 将这个思想称为 iteration batching / continuous batching：batch 在运行过程中可以动态改变，已完成请求立即返回，新请求可以加入后续迭代；相较 conventional batching，可在相同延迟约束下获得显著更高吞吐。[^friendli]

### 1.2 Orca 的两个关键词

Orca 的核心有两个概念：

| 概念 | 含义 | 对 inferlite M3 的启发 |
|---|---|---|
| Iteration-level scheduling | 调度器每次只调度一个 decode iteration，而不是整个 request 生命周期 | M3 的 decode loop 每轮重新从 `running` 请求组 batch |
| Selective batching | Transformer 里不是所有操作都适合无脑 batch，需要选择性 batch | M3 先只 batch decode，不 batch prefill |

Orca 论文摘要还提到 selective batching：为了同时应用 batching 与 iteration-level scheduling，需要只对部分操作进行 batching。[^orca]

inferlite M3 的“prefill 逐条、decode batch”就是一个教学版 selective batching：先把最关键、最容易讲清楚的 decode iteration batching 打穿。

### 1.3 vLLM 的工程化版本

vLLM V1 中，Engine Core 负责运行 scheduler、管理 KV cache、协调 GPU worker；API server 负责 HTTP 与输入处理，GPU worker 负责模型 forward 和显存管理。[^vllm-arch]

这说明生产系统会把 continuous batching 放在 engine core 的常驻循环里：

```text
API Server → Engine Core(Scheduler + KV Cache) → GPU Worker(Model Forward)
```

vLLM 的工程能力远超 M3：

- PagedAttention：KV Cache 按 block 分配，避免连续 slot 的空间浪费。
- Chunked prefill：长 prompt 可切块，避免 prefill 阻塞 decode。
- Prefix cache：相同前缀复用 KV。
- CUDA Graph / FlashAttention / fused kernels：降低 kernel launch 和显存带宽开销。
- 多进程 / 多 GPU / tensor parallel。

inferlite M3 不追求这些生产能力，而是只保留 continuous batching 的最小内核：

```text
waiting / running / finished 三队列
+ 每个 decode step 动态组 batch
+ slot-based KV Cache
+ 每请求独立 seq_len
```

## 2. inferlite M3 的目标与非目标

### 2.1 一句话目标

M3 要实现一个教学版 continuous batching：

> **Prefill 因 prompt 长短不一逐条做；Decode 因每步都是 1 个 token，可以按 iteration 动态组 batch。每个请求独占一个 KV slot，并维护自己的 `slot_id + seq_len`。**

M2 的心智模型是：

```text
一个请求
一个 KVCache
一个 cur_len
decode 每步只处理当前 token
```

M3 的心智模型是：

```text
多个请求
一个 BatchedKVCache
多个 slot
每个 slot 一个 seq_len
多个请求 decode step 合并成一次 batch forward
```

### 2.2 M3 主要收益来自哪里

M2 已经让单请求 decode 从重算完整历史变成只算当前 token：

```text
M2: 单请求 Attention 从 O(T²) 降到 O(T)
```

但如果有 8 个请求并发，M2 串行执行仍然会让每个请求单独读一遍权重：

```text
req1 decode step → model forward，读一次权重
req2 decode step → model forward，读一次权重
...
req8 decode step → model forward，读一次权重
```

M3 要变成：

```text
[req1, req2, ..., req8] decode step → 一次 model forward，读一次权重
```

所以 M3 的收益主要来自：

```text
多请求共享一次模型权重读取 / forward launch
```

而不是继续降低单请求 Attention 的渐进复杂度。

### 2.3 M3 不做什么

| 能力 | M3 是否支持 | 说明 |
|---|---:|---|
| 单请求 M2 generate | ✅ | 保持兼容 |
| 多请求提交 | ✅ | 支持多个 request |
| waiting / running / finished 三队列 | ✅ | L3 调度核心 |
| prefill 逐条做 | ✅ | 避免 batch prefill padding 复杂度 |
| decode batch | ✅ | M3 核心能力 |
| 每请求独立 cache length | ✅ | `seq_lens[slot]` |
| slot-based KV Cache | ✅ | 每请求独占一个连续 slot |
| 不同请求 EOS 早停 | ✅ | done 请求立即出队 |
| 新请求中途加入 | ✅ 简化支持 | 在 decode step 边界 admit |
| batch prefill | ❌ | 留后续 |
| chunked prefill | ❌ | 留 M11 |
| PagedAttention | ❌ | 留 M4 |
| Prefix / session cache | ❌ | 留 M5 |
| 有状态多轮 KV 复用 | ❌ | 留 M5 Prefix / Session Cache |
| 请求抢占 / preemption | ❌ | 留后续 |

## 3. 为什么 M3 只 batch decode，不 batch prefill

### 3.1 Batch prefill 的 padding 计算浪费严重

假设三个请求：

```text
reqA prompt_len = 100
reqB prompt_len = 300
reqC prompt_len = 80
```

如果 naive batch prefill，需要 padding 到最大长度 300：

```text
reqA: 100 token + 200 pad
reqB: 300 token
reqC:  80 token + 220 pad
```

真实 token 数：

```text
100 + 300 + 80 = 480
```

batch 后计算 token 数：

```text
3 × 300 = 900
```

浪费：

```text
420 / 900 ≈ 46.7%
```

更重要的是，prefill padding 浪费的不只是 KV 空间，还包括：

```text
embedding
q/k/v projection
RMSNorm
RoPE
Attention
MLP / FFN
lm_head
```

并且 prefill attention 是 `T × T`，padding 会放大 attention 计算。长度分布越不均匀，浪费越严重。

### 3.2 Decode batch 的浪费小得多

进入 decode 后，每个请求每步只输入 1 个 token：

```text
reqA 当前 token: [1]
reqB 当前 token: [1]
reqC 当前 token: [1]
```

所以可以拼成：

```text
input_ids shape = [B, 1]
```

decode 阶段没有 query token 的 padding 浪费，因为每行都是 1 个 token。

但读取 KV 时，由于每个请求历史长度不同，M3 会先按 batch 内最大长度读出：

```text
reqA seq_len = 100
reqB seq_len = 300
reqC seq_len = 80
max_len = 300
```

这会带来一些 O(T) 的 attention 浪费：短请求会读到并 mask 掉尾部无效 KV。但相比 prefill 的 O(T²) padding 浪费，这个代价小得多，M3 可以接受。

真正要消除这部分浪费，需要 M4 PagedAttention / ragged attention。

## 4. 请求组 batch 是否有时间限制

### 4.1 Continuous batching 不等于“等人凑满 batch”

M3 采用 opportunistic batching：

> **每个 decode iteration 边界，调度器拿当前已经 ready 的 running 请求组成 batch；不为了凑满 batch 无限等待。**

也就是说，batch 的形成由两个因素决定：

1. 当前有多少 running 请求。
2. `max_num_seqs` 还有多少容量。

如果当前只有 1 个请求 ready，就跑 batch size = 1；如果有 8 个请求 ready，就跑 batch size = 8。

M3 不做“等待 5ms 凑 batch”这类 serving 策略。原因是：

- 当前没有 HTTP server，没有真实在线到达时间。
- 教学目标是讲清 iteration-level scheduling，而不是延迟/吞吐 trade-off。
- 等待窗口会引入 wall-clock time、异步队列、超时、SLO 等复杂概念。

### 4.2 M3 的调度节奏

M3 的主循环可以理解为：

```python
while scheduler.has_unfinished():
    # 1. 如果有空 slot，则从 waiting admit 新请求
    admit_until_full()

    # 2. 对新 admitted 请求逐条 prefill
    prefill_new_requests_one_by_one()

    # 3. 对 running 请求做一次 batched decode
    decode_one_iteration_for_running_batch()

    # 4. EOS / max_new_tokens 的请求出队并释放 slot
    finish_done_requests()
```

这个循环没有“凑 batch 等待”。新请求最多等待到下一个 iteration 边界。

### 4.3 生产系统会有时间策略，但 M3 不做

生产系统可能会引入：

| 策略 | 作用 | M3 是否做 |
|---|---|---:|
| max waiting time | 请求排队太久则超时 | ❌ |
| batching window | 稍等几毫秒凑更大 batch | ❌ |
| SLO-aware scheduling | 延迟约束下最大化吞吐 | ❌ |
| preemption | 长请求让位给短请求 | ❌ |
| priority queue | 高优请求优先 | ❌ |
| max batched tokens | 防止单步 batch token 过多 | 简化为 `max_num_seqs` |

M3 只保留 `max_num_seqs` 作为容量约束。

## 5. KV Cache 是否有过期概念

### 5.1 M3 有释放，没有 session 级过期

M3 的 KV Cache 生命周期绑定到一次 request：

```text
submit request
  ↓
alloc slot
  ↓
prefill + decode
  ↓
EOS / max_new_tokens / error
  ↓
free slot
```

所以 M3 有明确的 slot 释放：

```python
kv_cache.free_slot(req.slot_id)
```

但 M3 没有 session cache / prefix cache，因此没有“跨请求 KV 过期”的概念。

换句话说：

```text
M3 的 KV 只在一个请求生命周期内有效
请求结束立即释放 slot
```

### 5.2 为什么 M3 不做 TTL

TTL 通常用于两类 cache：

| Cache 类型 | 是否跨请求 | 是否需要 TTL | 归属 |
|---|---:|---:|---|
| Request KV Cache | 否 | 通常不需要，结束即释放 | M3 |
| Session KV Cache | 是 | 需要，用户会话可能空闲 | M5+ |
| Prefix Cache | 是 | 需要 eviction / refcount / LRU | M5 / M4 后更自然 |

M3 的 slot 不是“可复用的历史上下文缓存”，而是 active request 的工作内存。因此它的释放条件不是时间过期，而是请求完成或取消。

### 5.3 如果请求异常怎么办

M3 可以预留取消接口：

```python
def cancel(request_id: str):
    req = find_request(request_id)
    req.status = CANCELLED
    kv_cache.free_slot(req.slot_id)
```

但第一版不需要做完整取消 API，只要在设计上明确：

```text
finished / cancelled / error 都必须释放 slot
```

## 6. 核心数据结构

### 6.1 RequestState

```python
@dataclass
class RequestState:
    request_id: str

    prompt_ids: torch.Tensor        # [1, T_p]
    generated_ids: list[torch.Tensor]

    max_new_tokens: int
    eos_token_id: int | None

    status: RequestStatus           # waiting / running / finished / cancelled

    slot_id: int | None             # 分配到哪个 KV slot
    seq_len: int                    # 当前 KV Cache 已写入长度
    num_generated: int              # 已采样 token 数

    last_token: torch.Tensor | None # 下一次 decode 要输入的 token
```

关键语义：

```text
seq_len    = KV Cache 已经写入的 token 数
last_token = 下一次 decode 要输入，但还没写进 cache 的 token
```

prefill 后：

```text
prompt_len = 100
seq_len = 100
prefill logits 采样得到 token_x
last_token = token_x
```

下一次 decode：

```text
输入 token_x
写入 cache[100]
seq_len = 101
采样得到 token_y
last_token = token_y
```

### 6.2 BatchedKVCache

M2 是：

```python
kv_cache.cur_len: int
```

M3 要变成：

```python
kv_cache.seq_lens: torch.Tensor  # [max_num_seqs]
```

结构：

```python
class BatchedKVCache:
    layers: list[LayerKVCache]
    seq_lens: torch.Tensor      # [max_num_seqs]
    occupied: torch.Tensor      # [max_num_seqs]
```

其中每层的 K/V：

```python
layer.k.shape = [max_num_seqs, n_kv_heads, max_seq_len, head_dim]
layer.v.shape = [max_num_seqs, n_kv_heads, max_seq_len, head_dim]
```

例子：

```text
slot 0: reqA, seq_len = 100
slot 1: reqB, seq_len = 300
slot 2: reqC, seq_len = 80
slot 3: empty
```

M3 的 slot 是 M4 PagedAttention 的简化版：每个请求独占一个连续 KV 区间。

### 6.3 FCFSScheduler

```python
class FCFSScheduler:
    waiting: deque[RequestState]
    running: dict[str, RequestState]
    finished: dict[str, RequestState]

    def submit(req): ...
    def admit(): ...
    def mark_finished(req): ...
```

核心不变式：

```text
len(waiting) + len(running) + len(finished) == total_requests
```

不做优先级，不做抢占。

## 7. 执行流程

### 7.1 请求提交

```python
scheduler.submit(reqA)
scheduler.submit(reqB)
scheduler.submit(reqC)
```

状态：

```text
waiting  = [reqA, reqB, reqC]
running  = []
finished = []
```

### 7.2 Admit + 单条 Prefill

```python
req = scheduler.admit()
slot = kv_cache.alloc_slot(req.request_id)

T_p = req.prompt_ids.shape[1]
position_ids = torch.arange(T_p).unsqueeze(0)

logits = model(
    input_ids=req.prompt_ids,       # [1, T_p]
    position_ids=position_ids,      # [1, T_p]
    kv_cache=kv_cache,
    cache_slots=torch.tensor([slot]),
    cache_positions=torch.tensor([0]),
)
```

prefill 写入：

```text
slot[0:T_p] = prompt 的 K/V
```

更新请求状态：

```python
req.slot_id = slot
req.seq_len = T_p
req.last_token = sampler(logits[:, -1, :])
req.num_generated = 1
req.status = RUNNING
```

注意：

```text
last_token 是第一个生成 token，但还没写入 KV Cache
```

### 7.3 Batched Decode

```python
running_reqs = [reqA, reqB, reqC]

input_ids = torch.cat([
    req.last_token for req in running_reqs
], dim=0)  # [B, 1]

cache_slots = torch.tensor([
    req.slot_id for req in running_reqs
])  # [B]

cache_positions = torch.tensor([
    req.seq_len for req in running_reqs
])  # [B]

position_ids = cache_positions[:, None]  # [B, 1]
```

一次 forward：

```python
logits = model(
    input_ids=input_ids,                 # [B, 1]
    position_ids=position_ids,           # [B, 1]
    kv_cache=kv_cache,
    cache_slots=cache_slots,             # [B]
    cache_positions=cache_positions,     # [B]
)
```

采样并逐请求更新：

```python
next_tokens = sampler(logits[:, -1, :])  # [B, 1]

for req, next_token in zip(running_reqs, next_tokens):
    req.seq_len += 1
    req.num_generated += 1
    req.last_token = next_token

    if next_token == req.eos_token_id or req.num_generated >= req.max_new_tokens:
        scheduler.mark_finished(req)
        kv_cache.free_slot(req.slot_id)
```

## 8. Decode Attention 的 mask 设计

### 8.1 物理结构

假设：

```text
reqA: slot=0, cache_position=100
reqB: slot=1, cache_position=300
reqC: slot=2, cache_position=80
```

当前 decode token 写入后：

```text
reqA valid KV = 0..100
reqB valid KV = 0..300
reqC valid KV = 0..80
```

为了组成 dense batch，读取 batch 内最大长度：

```python
max_len = max(cache_positions + 1)
```

这里：

```text
max_len = 301
```

读取：

```python
k = layer_cache.k[cache_slots, :, :max_len, :]
v = layer_cache.v[cache_slots, :, :max_len, :]
```

得到：

```text
k[0] = slot0[0:301]
k[1] = slot1[0:301]
k[2] = slot2[0:301]
```

但每行有效长度不同：

```text
reqA 只能看 0..100
reqB 只能看 0..300
reqC 只能看 0..80
```

### 8.2 Query 与 padding KV 的位置关系

每行的 query position 是：

```text
reqA q_pos = 100
reqB q_pos = 300
reqC q_pos = 80
```

对 reqA 来说：

```text
KV position 0..100   = 有效历史 + 当前 token
KV position 101..300 = padding / 无效 KV
```

也就是说：

```text
padding KV position > q_pos
```

它们位于 query 的未来位置之后，也不是该请求真实上下文的一部分，所以必须 mask。

### 8.3 当前 query 是否 attend to 自己

会，而且这是正确的。

decode 当前 token `x_t` 会产生：

```text
q_t, k_t, v_t
```

流程是：

```text
计算 q_t/k_t/v_t
写入 k_t/v_t 到 cache position t
读取 cache[0:t+1]
q_t attend to k_0..k_t
```

这等价于 full causal attention 中：

```text
位置 t 可以看历史 0..t-1，也可以看自己 t
不能看未来 t+1...
```

所以 mask 应该允许：

```python
k_pos <= q_pos
```

不能写成：

```python
k_pos < q_pos
```

否则当前 token 自己的 K/V 会被错误屏蔽。

### 8.4 Mask 代码

```python
valid_lens = cache_positions + 1
max_len = int(valid_lens.max().item())

positions = torch.arange(max_len, device=device)
visible = positions[None, :] < valid_lens[:, None]
# 等价于：
# visible = positions[None, :] <= cache_positions[:, None]
```

shape：

```text
positions: [max_len]
visible:   [B, max_len]
```

应用到 attention scores：

```python
# q: [B, n_q, 1, D]
# k: [B, n_q, max_len, D]
scores = torch.matmul(q, k.transpose(2, 3)) * scaling
# scores: [B, n_q, 1, max_len]

scores = scores.masked_fill(
    ~visible[:, None, None, :],
    torch.finfo(scores.dtype).min,
)
```

含义：

```text
scores[0, :, :, 101:301] = -inf
scores[1, :, :, :]       = 原值
scores[2, :, :, 81:301]  = -inf
```

## 9. Query 和自己的 K 为什么不会必然支配 attention

当前 token 的 query 会和自己的 key 做内积：

```text
q_t · k_t
```

但这不是“向量和自己点积”，因为：

```text
q_t = W_q h_t
k_t = W_k h_t
W_q != W_k
```

Q/K 是不同投影空间，语义角色不同：

```text
Q: 当前 token 想查什么
K: 当前 token 可以被如何索引
V: 当前 token 被关注时提供什么内容
```

同一个 token 的 q/k 位置编码相同，因此 RoPE 不改变 self 内积：

```text
RoPE(q_t, t) · RoPE(k_t, t) = q_t · k_t
```

但 self score 是否最大，取决于 `W_q h_t` 与 `W_k h_t` 的方向是否接近。模型可以学到当前 query 更关注历史关键 token。

如果忽略 RMSNorm weight 和 eps，RMSNorm 后每个 head 向量 L2 范数约为：

```text
||q||₂ = sqrt(d)
||k||₂ = sqrt(d)
```

所以内积上界：

```text
q · k ≤ d
```

经过 attention scaling：

```text
(q · k) / sqrt(d) ≤ sqrt(d)
```

对于 `head_dim=128`：

```text
最大 scaled score ≈ sqrt(128) ≈ 11.3
```

这只是理论上界，不表示 self score 一定最大。

## 10. 与 Orca / vLLM / nano-vLLM 的差异

| 维度 | Orca | vLLM | nano-vLLM | inferlite M3 |
|---|---|---|---|---|
| 调度粒度 | iteration-level scheduling | continuous batching / scheduler loop | 简化 scheduler | decode iteration |
| Prefill | selective batching，系统级优化 | prefill/decode 调度，支持更复杂策略 | 简化 | 逐条 prefill |
| Decode | batch decode | batch decode | batch decode | batch decode |
| KV 管理 | 系统级 KV 管理 | PagedAttention / block manager | block manager 简化版 | slot-based contiguous KV |
| 请求进入 | batch 运行中可动态加入 | scheduler 动态 admit | 简化 | decode step 边界 admit |
| 请求退出 | 结束立即返回 | 结束释放 blocks | 简化 | 结束释放 slot |
| 时间策略 | serving 系统考虑 latency/throughput | 有复杂调度参数 | 简化 | 不凑 batch，不等人 |
| Cache 过期 | 系统策略 | block refcount / eviction / prefix cache | 简化 | active request 结束即释放 |
| 目标 | 生产 serving | 生产 serving | 教学/极简高性能 | 教学：讲清 continuous batching 最小闭环 |

---

## 11. M3 与 M4 的边界

M3：

```text
slot-based KV Cache
每个请求独占一个完整连续 slot
实现简单
浪费显存空间
```

M4：

```text
PagedAttention
每个请求由多个 block 组成
按需分配 KV block
减少显存浪费
支持 Copy-on-Write / fork
```

M3 解决：

```text
多请求 decode 的权重 I/O 重复
```

M4 解决：

```text
KV Cache 空间浪费与碎片化
```

两者不要混在一个里程碑里做。

## 12. 验收目标

### 12.1 正确性

- 单请求 M2 路径保持不变。
- 多请求 batched decode 输出与逐条串行 M2 输出完全一致。
- 不同 prompt 长度的请求可以同 batch decode。
- 不同生成长度 / EOS 早停请求可以提前出队。
- slot 释放后可以复用，且旧 KV 不会泄漏。
- `seq_lens[slot]` 独立维护，互不影响。

### 12.2 性能

目标：

```text
8 并发请求聚合吞吐 ≥ 串行 M2 ×4
```

原因：

```text
M3 让多个请求共享一次模型权重读取
```

Mac MPS 可以先验证功能；最终性能数最好在 GPU 上补测。

## 13. 本版设计决策

| 决策 | 结论 | 原因 |
|---|---|---|
| Prefill 是否 batch | 否 | padding 计算浪费严重，且引入复杂 mask / chunked prefill |
| Decode 是否 batch | 是 | 每请求每步都是 1 token，天然适合组 batch |
| Decode KV 是否按 max_len 读取 | 是 | 实现简单，用 per-row mask 保证正确 |
| 每请求是否独占 slot | 是 | M2 静态 KV Cache 的自然扩展 |
| 是否等待凑 batch | 否 | M3 不做 batching window；在 iteration 边界即刻调度 |
| KV 是否有 TTL | 无 session TTL | request 完成 / 取消 / error 即释放 slot |
| 是否支持 PagedAttention | 否 | 留 M4 |
| 是否支持有状态多轮 KV 复用 | 否 | 留 M5 Prefix / Session Cache |
| 是否支持新请求中途加入 | 简化支持 | 在 decode step 边界 admit，prefill 后加入下一轮 decode batch |

---

## 14. 一句话总结

> **M3 不解决 KV Cache 空间浪费，只解决多请求 decode 的权重 I/O 重复。它不是等待凑满 batch，而是在每个 decode iteration 边界把当前 ready 的 running 请求动态组成 batch；KV Cache 生命周期绑定 active request，请求结束即释放 slot，不做跨请求 session TTL。实现上，prefill 因长度差异逐条计算；decode 因每步都是 1 token 可以组 batch；每个请求独立维护 `slot_id + seq_len`，Attention 用 `cache_slots + cache_positions + per-row mask` 保证每个 query 只看自己的有效历史。**
