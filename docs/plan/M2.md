# M2 — KV Cache

> **一句话目标**：把 M1 的"每步 full forward"改成"一次 prefill + N 次轻量 decode"，decode 阶段单步 Attention 复杂度从 O(T²) 降至 O(T)，同时支持 MPS/CUDA + bf16 推理。
>
> **完成日期**：2026-06-29
> **设计文档**：[m2-kv-cache-design.md](m2-kv-cache-design.md)（详细 ADR 记录）
> **实测结果**：[bench/results/2026-06-29-m2-kv-cache-mps-bf16.md](../../bench/results/2026-06-29-m2-kv-cache-mps-bf16.md)

## 1. 背景：M1 的浪费在哪里

M1 每个 decode 步都把完整历史序列过一遍 Attention：

```
decode 第 k 步（k = 1..N）：
  输入序列长度 = T_p + k
  每层 Attention：Q[T_p+k, D] @ K^T[T_p+k, D]，计算量 O((T_p+k)²)
```

N 步总 Attention 计算量约为 O(N·T²)（T = T_p + N，最终序列长度）。

**浪费的根源**：decode 第 k 步中，前 T_p+k-1 个 token 的 K/V 向量与第 k-1 步**完全相同**（causal mask 保证 K/V 只依赖当前及之前的 token，不受后续 token 影响），但 M1 每步都从头重算。

---

## 2. KV Cache 的核心原理

### 2.1 为什么可以缓存 K/V，不能缓存 Q

causal mask 的数学性质：位置 t 的 K/V 只依赖位置 0..t 的 token，在 decode 过程中永远不变。而 Q 代表"当前 token 想查什么"，每步生成的新 token 都不同，不可复用。

所以：**K/V 缓存，Q 每步重新计算。**

### 2.2 两个阶段

| 阶段 | 输入规模 | Attention 计算量 | 可缓存？ |
|------|---------|----------------|---------|
| **Prefill** | T_p 个 token 并行 | O(T_p²)，T_p × T_p 矩阵乘 | 否——token 间两两计算，无法跳过 |
| **Decode**（每步） | 1 个 token | O(T)，1 × T 向量乘 | **是**——历史 K/V 不受新 token 影响 |

KV Cache **只优化 decode 阶段**，不改变 prefill。

### 2.3 decode 步的新流程

缓存 K/V 后，decode 第 k 步（k = 1..N）：

```
1. 只计算当前 1 个 token 的 Q：shape [B, H_q, 1, D]
2. 读取 cache 中前 T_p+k-1 个 K/V：shape [B, H_kv, T_p+k-1, D]
3. Attention：Q[1 × (T_p+k-1)] 向量乘，计算量 O(T)
4. 把当前 token 的 K/V 追加写入 cache 第 T_p+k 位置
```

N 步总计算量降至 O(N·T)，比 M1 的 O(N·T²) 少了 T 倍。

### 2.4 实测验证

**Mac M3 Pro，MPS，bfloat16，Qwen3-0.6B，gen_tokens=128**：

| prompt_tokens | M1 tok/s | M2 tok/s | Speedup |
|--------------|---------|---------|---------|
| 32  | 13.8 | 24.9 | 1.80× |
| 64  | 12.7 | 28.9 | 2.27× |
| 128 |  9.6 | 25.7 | 2.67× |
| 256 |  6.2 | 24.1 | 3.91× |
| 512 |  3.3 | 24.1 | **7.36×** |

M2 的 tok/s 基本稳定（每步固定处理 1 个 token），M1 随 prompt 长度线性下降，符合 O(T²) vs O(T) 的理论预期。

**理论 ~600× vs 实测 7.36× 的差距**：每步 decode 还需把 28 层权重（~1.2 GB）从内存读一遍——这是 M1/M2 共有的权重 I/O 开销，KV Cache 无法消除。对 0.6B 小模型，权重读取远比 Attention 计算重，所以实测加速比远低于理论上限。模型越大、prompt 越长，实测加速比越接近理论值。

## 3. 数据结构设计

### 3.1 LayerKVCache + KVCache

```python
# inferlite/model/kv_cache.py

@dataclass
class LayerKVCache:
    k: torch.Tensor  # [B, n_kv_heads, max_seq_len, head_dim]
    v: torch.Tensor  # [B, n_kv_heads, max_seq_len, head_dim]

class KVCache:
    def __init__(self, layers: list[LayerKVCache]) -> None:
        self.layers = layers
        self.cur_len: int = 0  # 已写入多少个 token，唯一事实源

    @classmethod
    def from_config(cls, config, batch_size, max_seq_len, dtype, device) -> KVCache:
        # 按模型层数预分配所有层的 K/V tensor
        ...

    def reset(self) -> None:
        self.cur_len = 0  # tensor 不清零，下次 prefill 会覆盖写入
```

**为什么静态预分配**：decode 阶段零内存分配（切片写入不触发 `torch.cat`），避免每步申请新内存。代价是需要提前知道最大序列长度（`max_seq_len`）。

**`cur_len` 为什么放在 KVCache 而不是每层各自维护**：所有层共享同一个 `cur_len`，generate loop 显式更新，状态变化一目了然，不需要 trace 进 cache 内部。

### 3.2 KV Cache 内存布局

```
预分配 tensor：shape [1, H_kv, L_max, D]（单层，B=1）

位置索引：  0    1    2   ...  T_p-1  T_p  T_p+1 ...  L_max-1
           ┌─────────────────────┬─────┬──────┬──────────────┐
prefill后：│K_0  K_1  K_2 ... K_Tp-1│  0  │  0   │    ...       │
           └─────────────────────┘     ↑
                已写入 T_p 个         cur_len=T_p（未使用区域）

decode第1步后：
           │K_0  K_1 ... K_Tp-1  K_Tp│  0  │  0  │  ...         │
                                      ↑
                               新写入，cur_len → T_p+1

读取：full_k = k[:, :, :cur_len+1, :]  ← 切片只取有效范围，不含尾部零值
```

**Qwen3-0.6B 显存估算**（B=1，L_max=1024，bf16）：
```
2(K+V) × 1(B) × 28(层) × 1024(位置) × 8(KV heads) × 64(head_dim) × 2(bf16 bytes) = 58 MB
```

## 4. Attention 读写逻辑

### 4.1 新接口

```python
# inferlite/model/attention.py - GQAAttention.forward 新签名
def forward(
    self,
    hidden_states: torch.Tensor,
    position_embeddings: tuple[torch.Tensor, torch.Tensor],  # 由 Qwen3Model 统一计算传入
    layer_kv_cache: LayerKVCache | None = None,
    cache_position: int | None = None,  # = kv_cache.cur_len
) -> torch.Tensor:
```

### 4.2 写入与读取

```python
if layer_kv_cache is not None:
    # 写入当前 token(s)：切片赋值，原地写入，不分配新内存
    layer_kv_cache.k[:, :, cache_position:cache_position + T, :] = k
    layer_kv_cache.v[:, :, cache_position:cache_position + T, :] = v
    # 读取完整有效历史：只取 [0, cache_position+T) 范围，不含尾部零值
    k = layer_kv_cache.k[:, :, :cache_position + T, :]
    v = layer_kv_cache.v[:, :, :cache_position + T, :]
```

**为什么用切片而不是 `index_copy_`**：切片写入直接看代码就知道"写哪里、读哪里"，比 `index_copy_` + `cumulative_length` 更透明；且 MPS 上 `index_copy_` 有兼容性问题，切片无此限制。

### 4.3 causal mask：用 `T > 1` 判断

```python
if T > 1:  # prefill 阶段需要 causal mask，decode 步（T=1）不需要
    causal_mask = torch.triu(torch.ones(T, T_k, dtype=torch.bool), diagonal=1)
    attn_scores = attn_scores.masked_fill(causal_mask, -inf)
```

| 场景 | T | T > 1 | 需要 mask | 结论 |
|------|---|-------|---------|------|
| M1 full forward | prompt_len | ✓ | ✓ | 正确 |
| M2 prefill | prompt_len | ✓ | ✓ | 正确 |
| M2 decode 步 | 1 | ✗ | ✗ | 正确 |

**为什么不用 `if kv_cache is None` 判断**：prefill 阶段有 cache 但仍需 mask，此条件是错的。

## 5. Generate Loop 拆分

### 5.1 两条路径

```python
# inferlite/engine/core.py - generate()
def generate(..., kv_cache=None):
    if kv_cache is not None:
        # M2 路径：prefill + decode loop
        kv_cache.reset()

        # Prefill：一次处理完整 prompt
        position_ids = torch.arange(T_p).unsqueeze(0)   # [[0, 1, ..., T_p-1]]
        logits = model(input_ids, position_ids=position_ids, kv_cache=kv_cache)
        kv_cache.cur_len = T_p                           # ← 显式更新
        next_token = sampler(logits[:, -1, :])

        # Decode loop：每步只传 1 个 token
        for step in range(max_new_tokens - 1):
            pos = torch.tensor([[kv_cache.cur_len]])     # 绝对位置
            logits = model(next_token, position_ids=pos, kv_cache=kv_cache)
            kv_cache.cur_len += 1                        # ← 显式更新
            ...
    else:
        # M1 路径：full forward（原逻辑不变，向后兼容）
        ...
```

### 5.2 position_ids 必须用绝对位置

decode 步 `position_ids = [[cur_len]]`，不能写 `[[0]]`。

RoPE 的正确性依赖：`q_m · k_n` 编码相对距离 (m-n)。若 decode 每步都用 `[[0]]`，则每步都认为当前 token 在位置 0，与历史 K 的距离信息全部错误，生成质量严重下降，且**不报错**（沉默 bug）。

### 5.3 cur_len 的推进时序

```
kv_cache.reset()              → cur_len = 0

prefill:
  写入位置 [0, T_p)           → cur_len 仍为 0（写入时 cache_position=0）
  kv_cache.cur_len = T_p      → cur_len = T_p

decode step 1:
  写入位置 [T_p, T_p+1)       → cache_position = cur_len = T_p
  kv_cache.cur_len += 1       → cur_len = T_p+1

decode step 2:
  写入位置 [T_p+1, T_p+2)     → cache_position = cur_len = T_p+1
  kv_cache.cur_len += 1       → cur_len = T_p+2
  ...
```

## 6. 与 transformers / nano-vllm 的对比

| 维度 | transformers StaticCache | nano-vllm/vLLM | inferlite M2 |
|------|--------------------------|----------------|--------------|
| 内存策略 | 预分配连续 tensor | 全局预分配，block 粒度动态分配 | 预分配连续 tensor |
| 写入方式 | `index_copy_` 原地写 | Triton kernel，slot 散列写 | 切片写入（直观，MPS 兼容） |
| 读取方式 | 返回完整 tensor（含零值尾部） | FlashAttn + block_table 间接读 | 切片读有效范围 |
| `cur_len` 管理 | 藏在 cache 对象内部（`cumulative_length`） | 调度器管理，外部不感知 | generate loop 显式维护 |
| 多序列 | 需 padding mask | 天然支持 continuous batching | 不支持（M3 范围） |
| 依赖 | 纯 PyTorch | Triton + FlashAttention | 纯 PyTorch |

inferlite M2 与 transformers StaticCache 最接近，差异在于：切片写入替代 `index_copy_`（MPS 兼容 + 更可读），`cur_len` 外置显式维护（状态可见）。

## 8. 踩坑记录

| 坑 | 现象 | 根因 | 修复 |
|----|------|------|------|
| `position_ids = [[0]]` | 推理不报错，输出乱序 | decode 步 RoPE 编码全在位置 0，相对距离信息丢失 | `[[cur_len]]` 绝对位置 |
| causal mask 用 `if kv_cache is None` | prefill 有 cache 时 mask 丢失 | prefill 和 decode 的区分是 T>1，不是有无 cache | 改为 `if T > 1` |
| `reset()` 是否清零 tensor | 不清零也没问题 | prefill 会覆盖写入，旧数据不影响 | `reset()` 只清 `cur_len`，无需 `fill_(0)` |

## 9. 下一步：M3 Continuous Batching

M2 留下三个接口桩，M3 在此基础上生长：

1. **`KVCache` 是单序列设计**（`cur_len` 一个），M3 需要多序列各自维护状态，或换成 PagedAttention block table。
2. **`generate()` 是单请求循环**，M3 的 `FCFSScheduler` 把多个请求合并进同一个 `forward`，共享一次权重读取——这才是吞吐量提升的真正来源（单序列 KV Cache 解决的是计算量，continuous batching 解决的是 I/O 利用率）。
3. **device/dtype 支持**（M2 T5 已完成）是 M3 的先决条件，可直接复用。
