## 1. 一句话结论

把 M1 的"每步 full forward"改成"一次 prefill + N 次轻量 decode"，decode 阶段单步 Attention 复杂度从 O(T²) 降至 O(T)，prompt=512 时实测加速 **7.36×**，同时支持 MPS/CUDA bf16 推理。

---

## 2. 实测结果

**环境**：Mac M3 Pro，MPS，bfloat16，Qwen3-0.6B，gen_tokens=128

| prompt_tokens | M1 tok/s | M2 tok/s | Speedup |
|--------------|---------|---------|---------|
| 32  | 13.8 | 24.9 | 1.80× |
| 64  | 12.7 | 28.9 | 2.27× |
| 128 |  9.6 | 25.7 | 2.67× |
| 256 |  6.2 | 24.1 | 3.91× |
| 512 |  3.3 | 24.1 | **7.36×** |

**规律**：M2 的 tok/s 基本稳定在 24-29（每步固定处理 1 个 token），M1 随 prompt 线性下降。加速比随 prompt 长度近线性增长，符合 O(T²) vs O(T) 的理论预期。

**理论 vs 实测的差距**：理论加速比可达 T 倍（~600×），实测 7.36× 的原因是每步 decode 还需把 28 层权重（~1.2 GB）从内存读一遍——这是 M1/M2 共有的权重 I/O 瓶颈，KV Cache 无法消除。M3 Continuous Batching 多请求共享一次权重读取后，收益会更显著。

## 3. 核心设计决策回顾

### 3.1 静态预分配 + `cur_len` 作为唯一事实源

预分配 28 层 × `[B, n_kv, max_seq_len, head_dim]` 的 tensor，decode 阶段零内存分配（切片写入，不触发 `torch.cat`）。`KVCache.cur_len` 是唯一状态，`generate()` 里显式维护，状态变化一目了然，不需要 trace 进 cache 内部。

对比 transformers 的选择：DynamicCache 每步 `torch.cat` 更灵活但有内存开销；StaticCache 预分配但状态（`cumulative_length`）藏在 cache 对象里。inferlite 选择"状态外置"，牺牲封装性换可读性。

### 3.2 切片读写替代 `index_copy_`

写入用 `kv.k[:, :, pos:pos+T, :] = k`，读取用 `kv.k[:, :, :pos+T, :]`——直接看代码就知道"写哪里、读哪里"，比 `index_copy_` + `cumulative_length` 更透明，且 MPS 兼容。

### 3.3 `position_ids` 必须用绝对位置

decode 步每次只处理 1 个 token，`position_ids` 必须是 `[[cur_len]]`（绝对位置），不能写 `[[0]]`。写 `[[0]]` 是沉默 bug：不报错，但每步都认为当前 token 在序列开头，RoPE 失效，生成内容乱序。

### 3.4 `T > 1` 判断 causal mask

prefill 需要 causal mask（prompt 内部的因果性），decode 步 T=1 不需要（当前 token 只看历史，没有"未来"）。用 `if T > 1` 是正确的，用 `if kv_cache is None` 是错的——prefill 阶段有 cache 但仍需 mask。

### 3.5 向后兼容：`kv_cache=None` 退化为 M1

M1 的 95 个单测全部继续通过，`kv_cache=None` 时走原 full forward 路径，不改动任何 M1 逻辑。这个设计让"有 cache == 无 cache 的输出"成为可验证的正确性断言（`torch.equal`）。

## 4. 踩坑与教训

| 坑 | 现象 | 根因 | 教训 |
|----|------|------|------|
| `position_ids = [[0]]` | 推理不报错，但输出乱序 | decode 步 RoPE 编码全部在位置 0，相对距离信息丢失 | decode 步 `position_ids` 必须是绝对位置，`[[cur_len]]` 而非 `[[0]]` |
| causal mask 用 `if kv_cache is None` 判断 | prefill 有 cache 时 mask 丢失，logits 错误 | prefill 和 decode 的区分不是"有没有 cache"，而是"序列长度是否 > 1" | mask 逻辑用 `if T > 1`，和 cache 状态解耦 |
| `reset()` 是否需要清零 tensor | 不清零，prefill 会覆盖写入，没有问题 | `cur_len` 是唯一状态，reset 只需清 `cur_len = 0` | 静态预分配 tensor 的旧数据会被覆盖，不需要 `fill_(0)` |
| ModelScope 缓存路径 | `~/.cache/modelscope/hub/models/Qwen/Qwen3-0___6B`（`.` 变 `___`） | ModelScope 的文件系统安全替换 | benchmark 和 CLI 文档里要用实际路径，不能用 `0.6B` |
| `AutoTokenizer.from_pretrained` 报 HFValidationError | 本地路径被当成 HF repo id 验证 | transformers 5.x 的 path/repo id 校验变严 | 本地路径必须加 `local_files_only=True` |

## 5. 学到了什么

**关于推理机制**

- **prefill 和 decode 是两种本质不同的计算**：prefill 是 T_p 个 token 并行算 Attention（O(T_p²)），不可绕过；decode 是每步 1 个 token 查历史（O(T)），可以用 cache 加速。理解这个分界才能理解 KV Cache 为什么只优化后半段。
- **GQA（Grouped Query Attention）是 KV Cache 的天然节约**：Qwen3-0.6B 中 H_q=16、H_kv=8，每 2 个 Q head 共享 1 对 KV，cache 显存直接减半。GQA 不是独立优化，而是和 KV Cache 协同设计的。
- **权重 I/O 才是小模型的真正瓶颈**：对于 0.6B 模型，每步 decode 的 Attention 计算量很小，但读取 28 层权重（~1.2 GB）是固定开销。KV Cache 节省的是 Attention 的 O(T²) 计算，不能消除权重读取。模型越大、batch 越多，KV Cache 的相对收益越高。

**关于实现**

- **"状态外置"vs"状态内置"是一个工程取舍**：inferlite 选择把 `cur_len` 放在 `generate()` 里显式维护，而不是藏在 cache 对象里。前者可读性高（状态变化一眼看到），后者封装性好（调用方不感知）。教学框架优先可读性。
- **`time.perf_counter` + warmup 是 benchmark 的标配**：不 warmup 的第一次跑包含 MPS JIT 编译开销，数据严重偏高。MPS/CUDA 都需要 warmup，CPU 不严格需要但 warmup 无害。
- **单测和 benchmark 不能混用**：单测要求秒级、无副作用、可在 CI 跑；benchmark 需要真实模型、足够长的序列、多次采样。两者目的不同，分开维护。

## 7. 下一步：M3 Continuous Batching 的连接点

M2 留下了三个明确的"接口桩"，M3 要在这些桩上生长：

1. **`KVCache` 是单序列设计**（`batch_size=1`，单一 `cur_len`）。M3 需要改为多序列各自维护 `cur_len`，或者换成 PagedAttention 的 block table。

2. **`generate()` 是单请求循环**。M3 的 `FCFSScheduler` 要把多个请求合并进同一个 `forward` 调用，共享一次权重读取——这才是吞吐量提升的真正来源。

3. **`--device mps/cuda` 和 `dtype bf16` 是 M3 的先决条件**。M2 T5 已经完成，M3 可以直接复用。
